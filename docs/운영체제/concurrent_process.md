# 병행 프로세스
프로세스 여러 개가 동시에 실행되는 것을 병행 프로세스라고 부른다. 

> 💡 병행성  
> 여러 개의 프로세스 또는 쓰레드가 동시 수행되는 시스템의 특성

## 병행 프로세스  실행 형태
**1. 인터리빙 형식**
- 인터리빙 (Interleaving) 형식은 메모리 주소의 일부를 각각의 메모리 모듈로 나누어 순차적으로 접근하는 방식
- CPU가 메모리에 접근할 때 여러 메모리 모듈에 걸쳐 균등하게 접근하는 것처럼 보이지만, 실제로는 하나의 CPU 코어만 작동하며 메모리 모듈에 차례대로 접근함
![Untitled](https://github.com/cyd5538/cyd5538.github.io/assets/91642972/09129471-7c83-426d-8893-33622048b59f)

**2. 병럴처리 형식**
-  여러 개의 CPU가 동시에 여러 프로세스를 처리합니다. 이를 병렬 처리라고 합니다.
-  공유 메모리 구조와 분산 메모리 구조로 나뉨
![Untitled](https://github.com/cyd5538/cyd5538.github.io/assets/91642972/9391bce2-5d32-4e56-a9d0-dc2c1d6062d3)

**공유 메모리 구조**
- 모든 프로세서가 단일 메모리 공간에 접근
- 프로세서 간의 통신이 빠르고 간단합니다. 모든 프로세서가 메모리를 공유하므로 데이터 공유가 쉽습니다.
- 대표적으로 대칭적 다중 처리기(Symmetric Multiprocessing, SMP) 시스템이 이에 해당

**분산 메모리 구조**
- 각 프로세서가 독립적인 메모리 공간을 가짐
- 이러한 시스템에서는 프로세서 간의 통신이 상대적으로 느리고 복잡
- 대표적으로 비대칭 다중 처리기(Asymmetric Multiprocessing, ASMP) 시스템이 이에 해당

## 프로세스 간의 관계

**독립 프로세스**
- 다른 프로세스의 영향을 받지 않고 독립적으로 실행되는 프로세스입니다.
- 다른 프로세스와 데이터 또는 상태를 공유하지 않습니다.
- 실행 결과가 항상 입력에 의해 결정되므로 예측 가능
- 다른 프로세스의 영향을 받지 않기 때문에 디버깅이 용이

**협력 프로세스**
- 서로 영향을 주고받으며 협력하여 작업을 수행하는 프로세스
- 데이터 또는 상태를 공유하여 정보를 교환하고 작업을 분담
- 프로세스 간 통신이 용이하고 데이터 공유가 간편하여 효율성이 높음
- 한 프로세스의 오류가 다른 프로세스에 영향을 미칠 수 있다

## 병행성 문제
협력 프로세스는 서로 정보를 공유하고 작업을 분담하여 효율성을 높일 수 있지만, 동시에 병행성 문제라고 불리는 여러 가지 문제가 발생할 수 있다.

**1. 상호 배제**
두 개 이상의 프로세스가 동시에 임계 영역을 수행하지 못하도록 제어하는 방식
> 💡 임계 영역   
> 공유 자원에 접근하거나 상태를 변경하는 프로그램 코드 영역

**2. 동기화**
두 개 이상의 프로세스가 적절한 순서로 실행되도록 조율하는 방식
- 프로세스 동기화: 프로세스가 서로 작업 순서를 조율하는 동기화

**3. 프로세스 간 통신 (IPC)** 
두 개 이상의 프로세스가 정보를 교환하는 방식
- 통신 방법으로는 하나의 변수를 사용하여 직접적으로 데이터를 공유하는 방법이나 메시지를 주고받는 방법 

## 세마포어
세마포어는 운영 체제에서 공유 자원에 대한 접근을 제어하는 데 사용되는 동기화 도구입니다. 
쉽게 말해서, 여러 프로세스가 동시에 같은 자원을 사용하는 것을 방지하여 경쟁 조건을 발생하지 않도록 합니다.
- 다익스트라가 제안한 교착 상태에 대한 해법
- 공유 자원의 사용 가능한 개수를 나타내는 정수 변수
- 세마포어는 P 연산과 V 연산이라는 두 가지 기본 연산을 사용하여 작동
- 세마포어마다 대기 큐 필요
- 
**P 연산**
- 프로세스가 자원에 접근하기 전에 수행하는 연산입니다.
- 세마포어 값을 감소시킵니다.
- 세마포어 값이 0이 되면 프로세스는 블록되어 자원이 사용 가능해질 때까지 기다립니다.

**V 연산**
- 프로세스가 자원 사용을 마친 후에 수행하는 연산입니다.
- 세마포어 값을 증가시킵니다.
- 기다리고 있는 프로세스가 있다면 블록된 상태에서 벗어나 자원을 사용할 수 있도록 합니다.

### 상호배제 해결
- 세마포어 mutex 초깃값은 1
- 진입영역: P(mutex)
- 해제영역: V(mutex)
- 대기 큐는 FIFO로 동작
```
세마포어 mutex를 정의하고 초기값을 1로 설정한다.
FIFO 큐를 정의한다.

진입영역:
    P(mutex); // 세마포어 값을 감소시킨다.
    // 대기 큐를 사용하여 다른 프로세스가 임계 영역에 접근하는 것을 방지한다.

임계영역:
    // 임계영역의 코드를 수행한다.
    // 이 부분은 한 번에 하나의 프로세스만이 접근할 수 있는 임계 영역이다.

해제영역:
    // 다른 프로세스가 임계 영역을 수행할 수 있게 처리하는 부분이다.
    // FIFO 큐에서 대기 중인 프로세스를 꺼내어 실행한다.
    V(mutex); // 세마포어 값을 증가시킨다.
```

## 세마포어를 활용한 병행 프로세스 문제 해결
### 생산자 소비자 문제

생산자 소비자 문제와 세마포어를 이용한 해결 방식
1. 생산자 소비자 문제 개요
생산자 소비자 문제는 두 개의 협력 프로세스, 생산자와 소비자 사이의 상호작용을 다루는 고전적인 병행 처리 문제입니다.
생산자는 데이터를 저장하고 소비자는 데이터를 꺼내는 역할을 합니다.
생산자는 버퍼가 가득 차 있으면 기다려야 하고, 소비자는 버퍼가 비어 있으면 기다려야 합니다.

```
// 생산자
while (true) {
    p(empty); // 버퍼 사용 가능여부 확인
    p(mutex)
    // 임계영역 진행 - 데이터 생산
    ...
    v(mutex);
    v(full);
}

//소비자
while (true) {
    p(full); // 버퍼 사용 가능여부 확인
    p(mutex)
    // 임계영역 진행 - 데이터 소비
    ...
    v(mutex);
    v(empty);
}
```

### 판독기-기록기 문제
판독기-기록기 문제는 여러 판독기 프로세스가 공유 데이터를 읽는 반면, 단일 기록기 프로세스가 공유 데이터를 쓰는 상황을 다루는 병행 처리 문제입니다
- 판독기: 데이터를 읽는 프로세스
- 기록기: 데이터를 쓴느 프로세스

**상호 배제 구현**
- 기록기가 데이터를 쓰는 동안 판독기가 데이터를 읽지 못하도록 상호 배제를 구현
- 기록기가 데이터를 쓰지 않는 경우 여러 판독기가 동시에 데이터를 읽을 수 있도록 독자의 우선순위를 고려
- 판독기가 지속적으로 데이터를 읽는 경우 기록기가 공유 데이터를 영원히 쓰지 못하는 기아 상태를 방지

판독기의 우선순위를 두거나, 기록기의 우선순위를 두는 2가지 방식이 있다.

**판독기에 우선순위를 두었을 때**  
- 판독기는 즉시 공유자원 접근 가능
- 여러 판독기가 동시에 데이터를 읽을 수 있어 데이터 접근성이 향상되지만 기록기의 기아 상태에 빠질 가능성이 높다.
```
// 기록기
세마포어 wrt 초기값을 1로 설정 // 기록기 우선순위를 위한 세마포어
세마포어 mutex 초기값을 1로 설정 // 임계 영역에 진입하기 위한 세마포어
정수 rcount 초기값을 0으로 설정 // 판독기의 개수를 나타내는 변수

// 기록기
while (true) {
    p(wrt) // 기록기 우선순위를 위한 세마포어 획득
    // 임계 영역 진입 - 데이터 기록
    ...
    // 임계 영역 종료
    v(wrt) // 기록기 우선순위를 위한 세마포어 해제
}

// 판독기
while (true) {
    p(mutex) // 임계 영역에 진입하기 위한 세마포어 획득
    rcount++ // 판독기 개수 증가
    if (rcount == 1) {
        p(wrt) // 기록기가 데이터를 쓰는 동안에는 판독기 대기
    }
    v(mutex) // 임계 영역에 대한 세마포어 해제

    // 임계 영역 진입 - 데이터 판독
    ...

    p(mutex) // 임계 영역에 진입하기 위한 세마포어 획득
    rcount-- // 판독기 개수 감소
    if (rcount == 0) {
        v(wrt) // 모든 판독기가 종료되면 기록기에게 우선순위 부여
    }
    v(mutex) // 임계 영역에 대한 세마포어 해제
}

```
**기록기에 우선순위를 두었을 때**  
- 대기 중인 기록기가 있으면 공유 자원에 접근이 불가능하다
- 판독기의 병행성이 떨어지고, 기아 상태 유발 가능성
```
세마포어 mutex 초기값을 1로 설정 // 임계 영역에 진입하기 위한 세마포어
세마포어 wrt 초기값을 1로 설정 // 기록기가 데이터를 쓰고 있는지를 나타내는 세마포어
세마포어 rd 초기값을 1로 설정 // 판독기가 데이터를 읽고 있는지를 나타내는 세마포어

// 기록기
while (true) {
    p(wrt); // 데이터를 쓰는 동안에는 판독기가 접근하지 못하도록 함
    // 임계 영역 진입 - 데이터 기록
    ...
    // 임계 영역 종료
    v(wrt); // 데이터를 쓴 후에는 다시 판독기에게 접근을 허용함
}

// 판독기
while (true) {
    p(rd); // 데이터를 읽는 동안에는 기록기가 접근하지 못하도록 함
    // 임계 영역 진입 - 데이터 읽기
    ...
    // 임계 영역 종료
    v(rd); // 데이터를 읽은 후에는 다시 기록기에게 접근을 허용함
}
```

## 프로세스 간 통신(IPC)
- 병행 프로세스가 데이터를 공유하는 방법
    - 공유 메모리 방법
    - 메시지 전달 방법
하나의 운영체제에서 두 방법 함께 사용 가능하다.

### 공유 메모리 방법
협력 프로세스가 동일한 변수를 사용합니다.
  - 동일한 변수: 공유자원인 메모리 공간 사용
  - 생산자-소비자 문제의 유한 버퍼
  - 판독기-기록기 문제의 공유자원
- 대량 데이터 교환: 고속 통신 가능
- 통신상 발생 가능 문제 해결

**장점**
빠른 속도: 메시지 전달 방식보다 데이터 전송 속도가 빠름
높은 효율성: 대용량 데이터 전송 시 CPU 및 네트워크 자원 사용량 감소
간편한 구현: 복잡한 메시지 전달 프로토콜 없이 간편하게 구현 가능
  
**단점**
보안 문제: 여러 프로세스가 동시에 접근하여 데이터 경쟁, 손상, 보안 취약점 발생 가능성
동기화 문제: 데이터 손상 방지를 위한 동기화 메커니즘 필요 (뮤텍스, 세마포어 등)
메모리 관리 어려움: 공유 메모리 영역 할당 및 관리 복잡

**사용 예시**
- 생산자-소비자 문제: 생산자 프로세스가 생산한 데이터를 소비자 프로세스가 공유 버퍼를 통해 소비하는 경우
- 판독기-기록기 문제: 여러 판독기 프로세스가 데이터를 읽고, 기록기 프로세스가 데이터를 쓰는 경우

### 메시지 전달 방법
- 협력 프로세스가 메세지를 주고받음
    - 시스템 호출 send(), receive()
- 소량 데이터 교환에 적합
- 통신상 발생 가능 문제 해결 (운영체제)

논리적 구조  

통신 링크
- 메시지가 지나다니는 통로
- 
통신 링크 구현 형태
- 연결 대상: 협력 프로세스 서로
- 두 프로세스 사이 링크 개수 : 1개 이상
- 방향성 : 단방향, 양방향
- 통신 링크 용량 : 무한, 유한, 0

통신링크 용량
- 무한 : 통신링크 큐 존재, 송신자 대기 없음
- 유한 : 큐 존재, 큐가 가득 차면 송신자가 대기
- 0 : 큐가 없음, 수신자가 메시지를 받을 수 있을 때까지 송신자 대기

간접통신
- 프로세스 사이에 우편함을 통해 메시지 전달
- 같은 우편함을 이용하는 통신링크 설정
- 여러 우편함을 이용하면 여러 통신링크 존재
- 하나의 통신링크가 여러 프로세스와 연관 가능
- 단방향 또는 양방향

우편함이 수신 프로세스에 소속
- 하나의 수신자만 있는 경우에 사용됩니다. 이 경우 통신 링크는 단방향

우편함이 운영체제에 소속
- 여러 수신자가 있을 때 사용
- 운영체제는 수신자를 관리하고, 한 번에 하나의 수신자만이 메시지를 받을 수 있다. 
- 통신 링크는 양방향

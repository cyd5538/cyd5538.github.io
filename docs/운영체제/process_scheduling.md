# 프로세스 스케줄링(Process Scheduling)
- 주어진 프로세스가 여러 개인 경우, 프로세스 처리순서를 결정하는 것

## 스케줄링 단계
[이미지출처](https://danidanee.tistory.com/10)![스케줄링단계](https://github.com/cyd5538/cyd5538.github.io/assets/91642972/9fcd062d-79a0-42a4-b098-b3824f96b631)  
- **고수준 스케줄링**
  - 고수준 스케줄링은 시스템 전체의 작업량을 조절하는 역할
  - 시스템에 들어오는 작업 요청을 검토하여 승인할지 거부할지 결정
  - 작업의 입출력 요청을 처리하고, 입출력 장치를 공유합니다
- **중간 수준 스케줄링**
  - 전체 시스템의 활성화된 프로세스 수를 조절하여 과부하를 막음 
  - 각 작업에게 CPU 사용 시간을 할당
- **저수준 스케줄링**
  - 실제로 작업을 수행
  - 어떤 프로세스에 CPU를 할당할지, 어떤 프로세스를 대기 상태로 보낼지등을 결정

## 스케줄링 목적
- **공정성** : 모든 프로세스가 자원을 공정하게 배정받아야 하며, 자원 배정 과정에서 특정 프로세스가 배제당하면 안됩니다.
- **효율성** : 시스템 자원이 최대한 낭비하는 시간이 없이 활용되도록 스케줄링해야 됩니다
- **우선 순위** : 중요 프로세스가 먼저 작동하도록 우선순위를 정해주어야 합니다.
- **응답시간 보장** : 사용자는 시스템의 응답이 없으면 시스템이 멈춘 것으로 인지하기 때문에 응답 시간을 보장해야 합니다.

## 선점형, 비선점형 스케줄링
### 선점형
- 선점형 스케줄링은 현재 실행 중인 작업을 중단하고 더 중요한 작업을 먼저 실행하는 방식입니다. 즉, 작업 실행 도중에 우선순위가 높은 다른 작업이 나타나면 현재 작업을 중단시키고 새로운 작업을 실행하게 됩니다.
- 하나의 프로세스가 CPU를 독점할 수 없기 때문에 빠른 응답 시간을 요구하는 대화형 시스템이나 시분할 시스템에 적합
 
**장점**  
1. 중요한 작업을 먼저 처리하기 때문에 시스템 응답 시간을 단축
2. 실시간 응답이 중요한 시스템에서 유용

**단점**  
1. 문맥 교환에 따른 오버헤드 발생
2. 우선 순위가 낮은 작업은 계속 후순위로 밀릴 수 있음

### 비선점형
- 작업이 시작되면 종료될 때까지 중단하지 않고 계속 실행하는 방식 즉, 현재 작업이 종료되거나 자원을 반납할 때까지 다른 작업은 실행될 수 없습니다.
  
**장점**  
1. 모든 작업이 동일한 실행 시간을 보장
2. 문맥 교환에 의한 오버헤드가 적음

**단점**  
1. 긴 프로세스가 실행 중이라면 짧은 프로세스가 오래 기다리게 되는 경우 발생
2. 중요한 작업이 있어도 현재 작업이 종료될 때까지 기다려야 함

### 정리
| 구분             | 선점형 스케줄링                                         | 비선점형 스케줄링                                    |
|----------------|-------------------------------------------------------|----------------------------------------------------|
| 정의            | 현재 CPU를 점유하고 있는 프로세스가 더 높은 우선순위를 가진 다른 프로세스에 의해 강제로 중단될 수 있는 방식 | 현재 CPU를 점유하고 있는 프로세스가 자발적으로 종료하거나 자원을 반납할 때까지 다른 작업이 실행될 수 없는 방식 |
| 장점            | 응답 시간 짧음,시스템 성능 높음                    | - 공정성 높음, 예측 가능성 높음, 오버헤드 낮음     |
| 단점            | 공정성 낮음,예측 가능성 낮음, 오버헤드 높음   | - 응답 시간 길음, 시스템 성능 낮음                  |
| 적합한 시스템   | 실시간 시스템, 시분할 시스템                              | 배치 처리 시스템, 데이터베이스 시스템, 일반 운영 체제     |

## 스케줄링 알고리즘
스케줄링 알고리즘은 시스템 자원 (CPU, 메모리, 입출력 장치 등)을 여러 프로세스에 어떤 순서로 할당할지 결정하는 방법

### 스케줄링 알고리즘의 선택 기준
1. CPU 사용률
  - 전체 시스템의 동작 시간 중 CPU가 사용된 시간을 측정
  - 높은 CPU 사용률은 시스템 자원을 효율적으로 활용하고 있다고 알 수 있음
2. 처리량
  - 단위 시간당 작업을 완료한 프로세스의 수
  - 이 수치가 클수록 더 나은 알고리즘으로 간주됨

### 스케줄링 알고리즘 평가 기준
- 평균 대기 시간
  - 모든 프로세스의 대기 시간을 합한 뒤 프로세스의 수로 나눈 값
- 평균 반환 시간
  - 모든 프로세스의 반환 시간을 합한 뒤 프로세스의 수로 나눈 값

### FCFS 스케줄링
- 비선점방식
- 준비 큐에 도착한 순서에 따라 디스패치

**장점**
- 가장 간단한 스케줄링 기법

**단점**
- 짧은 프로세스가 긴 프로세스를 기다리거나 중요한 프로세스가 나중에 수행될 수도 있음
    - 시분할 운영체제나 실시간 운영체제에서는 부적합
- 프로세스들의 도착순서에 따라 평균반환시간이 크게 변함

### SJF 스케줄링
- 예상 실행 시간이 가장 짧은 프로세스를 먼저 CPU에 할당하여 실행하는 방식
- 비선점방식
  
**장점**
- 작업이 짧은 순서대로 처리되므로 시스템의 응답성과 처리량이 향상
- 짧은 응답 시간이 중요한 실시간 시스템에 적합

**단점**
- 실제로는 먼저 처리할 프로세스의 CPU 시간을 예상할 수 없음
- 먼저 도착한 작업에 유리하게 작동하기 때문에 공정하지 않음
- 짧은 작업을 우선 처리하기 때문에 긴 작업은 계속 기다려야 함

### 라운드 로빈 스케줄링
- 선점방식
- 각 프로세스에 일정한 시간 슬라이스를 할당하고, 해당 시간이 지나면 다음 프로세스로 넘어가는 방식
- 시간 할당량 안에 종료하지 못한 프로세스는 준비 큐의 마지막에 배치됨

**장점**
- 모든 프로세스에게 동일한 기회를 제공하여 응답 시간을 빠르게 유지함
- 각 프로से스에 정해진 시간만큼 CPU를 사용할 수 있기 때문에 어떤 프로세스도 영원히 기다리는 일이 발생하지 않음

**단점**
- 짧은 시간 간격마다 프로세스 간에 CPU를 교환해야 하기 때문에 컨텍스트 스위칭(문맥 교환)이 많이 발생
- 평균 대기 시간은 짧은 시간 슬라이스로 인해 증가할 가능성이 높음

### HRN 스케줄링
- SJF 스케줄링에서 발생할 수 있는 아사 현상을 해결하기 위해 만들어진 비선점형 알고리즘
- 다음에 실행될 프로세스를 선택할 때 각 프로세스의 응답률을 고려
  - 응답률: (대기 시간 + 서비스 시간) / 서비스 시간
  - 예상실행시간이 짧을수록, 대기시간이 길수록 응답비율이 커짐

**장점**
- SJF의 단점 보완: 긴 시간만 있는경우 굶주림 문제가 발생할 수 있는데 HRN은 대기시간을 고려하여 긴 작업이라도 일정 시간 지나면 작업 기회를 줌
- 다른 알고리즘보다 공정한 기회를 제공

**단점**
- 실제로는 프로세스의 CPU 시간을 예상할 수 없음
- 구현이 복잠함

### 다단계 피드백 큐 스케줄링
- 여러 개의 우선순위 큐를 사용하여 프로세스를 스케줄링하는 방식
- 각 큐에는 다른 우선순위가 할당되며, 프로세스는 예상 실행 시간이나 I/O 사용량과 같은 기준에 따라 큐에 배정
- 각 큐는 라운드 로빈이나 FCFS등 독자적 스케줄링 사용 가능

**장점**
- 다양한 우선순위를 가진 프로세스들을 효과적으로 처리하여 공정성이 높음
- 높은 우선순위 큐에 있는 프로세스를 우선 처리하여 사용자 응답 속도를 빠르게 유지

**단점**
- 큐 관리, 프로세스 이동 등에 필요한 추가적인 작업으로 인해 오버헤드가 증가
- 구현이 복잠(여러 개의 큐를 관리하고 각 큐에 적절한 스케줄링 알고리즘을 선택해야 하기 때문에 구현이 다소 복잡)

재조정

재조정 이해하기​

리액트의 가상 DOM은 우리가 원하는 UI 상태의 청사진입니다. 리액트는 이 청사진을 가지고 재조정(Reconciliation)이라는 프로세스를 통해 주어진 환경에서 실제 DOM으로 변환합니다.

리액트의 핵심 아이디어는 UI를 선언적으로 표현하는 것입니다. 개발자는 "UI가 이렇게 보여야 한다"고 선언하면, 리액트는 실제 DOM을 그 상태와 일치시키는 작업을 담당합니다. 이 과정에서 가상 DOM과 재조정 알고리즘이 핵심 역할을 합니다.

import { useState } from 'react';

const App = () => {
  const [count, setCount] = useState(0);

  return (
    <div>
      <h1>Count: {count}</h1>
      <button onClick={() => setCount(count + 1)}>증가</button>
    </div>
  )
}

재조정 과정을 이해하기 위해 위의 코드를 살펴보겠습니다.
이 코드는 버튼을 클릭할 때마다 카운트가 증가하는 간단한 리액트 컴포넌트입니다. JSX는 컴파일 시점에 리액트 엘리먼트의 트리로 변환됩니다. App 컴포넌트가 실행되면 내부에 중첩된 엘리먼트들을 포함한 하나의 리액트 엘리먼트 트리를 반환합니다. 리액트 엘리먼트는 불변 객체로, UI의 특정 시점 스냅샷을 나타냅니다. 이것은 화면에 무엇이 표시되어야 하는지를 설명하는 가벼운 객체이며, 실제 DOM 노드가 아닙니다. 리액트 엘리먼트는 React.createElement 함수나 JSX 문법을 통해 생성되며, 위 예시 코드는 내부적으로 다음과 같이 변환됩니다

const App = () => {
  const [count, setCount] = useState(0);

  return React.createElement(
    'div',
    null,
    React.createElement('h1', null, `Count: ${count}`),
    React.createElement(
      'button',
      { onClick: () => setCount(count + 1) },
      '증가'
    )
  );
};

이렇게 생성된 리액트 엘리먼트 트리는 다음과 같은 객체 구조를 가집니다

{
  type: 'div',
  props: {
    children: [
      {
        type: 'h1',
        props: { children: `Count: ${count}` }
      },
      {
        type: 'button',
        props: {
          onClick: () => setCount(count + 1),
          children: '증가'
        }
      }
    ]
  }
}

이 객체 구조는 가상 DOM의 일부로, 실제 DOM과 직접적으로 연결되어 있지 않습니다. 초기 렌더링 시, 리액트는 이 전체 가상 DOM 트리를 분석하여 필요한 실제 DOM 노드를 생성합니다. 리액트의 효율성은 상태 변경 후 발생하는 업데이트 과정에서 두드러집니다. 버튼이 클릭되어 count 상태가 변경되면, 리액트는 다음과 같은 과정을 통해 DOM을 효율적으로 업데이트합니다

  1. 컴포넌트가 재렌더링되어 새로운 리액트 엘리먼트 트리 생성
  2. 이전 트리와 새 트리 간의 차이점 계산
  3. 변경된 부분만 실제 DOM에 적용

일괄 처리​

리액트는 재조정 과정에서 여러 상태 업데이트를 일괄 처리(batching)하여 성능을 최적화합니다. 이 메커니즘은 불필요한 렌더링을 방지하고 실제 DOM 조작을 최소화하여 어플리케이션의 응답성을 향상시킵니다. 이를 이해하기 위해 여러 상태 업데이트가 연속적으로 발생하는 컴포넌트를 살펴보겠습니다.

function Example() {
  const [count, setCount] = useState(0);

  const handleClick = () => {
    setCount(count + 1);
    setCount(count + 2);
    setCount(count + 3);
  };

  return (
    <div>
      <h1>Count: {count}</h1>
      <button onClick={handleClick}>증가</button>
    </div>
  );
}

예시에서는 handleClick 함수가 호출될 때마다 setCount가 세 번 호출됩니다. 리액트는 일괄 업데이트를 수행하기 때문에 count + 1을 따로 실행해 DOM을 세 번 업데이트하는 대신에 count + 3으로 DOM을 한 번만 업데이트합니다. DOM에 대한 가장 효율적인 일괄 업데이트를 계산하기 위해 리액트는 현재 가상 DOM 트리를 복제해 새로운 가상 DOM 트리를 생성하고 업데이트된 값을 적용합니다. 예시에서는 count가 업데이트되어 3이 됩니다. 이 트리는 현재 브라우저에 있는 트리를 재조정해 0을 3으로 바꿔야 합니다. 이제 리액트는 DOM을 일일이 세 번 업데이트하지 않고 새 가상 DOM 값 3을 사용해 DOM을 한 번만 업데이트하면 된다고 계산합니다. 일괄 처리의 핵심 이점은 React가 모든 상태 변경을 단일 렌더링 사이클로 통합하여 가상 DOM 비교와 실제 DOM 업데이트를 한 번만 수행한다는 점입니다.

기존 기술​

스택 재조정자​

예전 리액트는 렌더링에 스택 데이터 구조를 사용했습니다. 컴퓨터 과학에서 스택은 선형 데이터 구조로, LIFO(Last In, First Out) 원칙을 따릅니다. 예전에 사용되던 리액트의 재조정자는 스택 기반 알고리즘을 사용해 새 가상 트리를 이전 가상 트리와 비교하고 그에 따라 DOM을 업데이트했습니다. 스택 재조정자는 간단한 경우에는 잘 작동했지만, 애플리케이션 규모가 커지고 복잡해지면서 여러 가지 문제가 발생했습니다.

우선 예시를 보겠습니다.

import React, { useState } from 'react';

const initialState = {
  inputValue: '',
  isButtonEnabled: false,
};

// CPU 자원을 소모하는 작업
const ExpensiveComponent = () => {
  for (let i = 0; i < 1000000000; i++) {}
  return <div>비싼 컴포넌트</div>;
};    
const Form = () => {
  const [state, setState] = useState(initialState);

  const handleInputChange = (e) => {
    const value = e.target.value;
    setState((prevState) => ({
      ...prevState,
      inputValue: value,
      isButtonEnabled: value.length > 0,
    }));
  };

  return (
    <div>
      <input type="text" value={state.inputValue} onChange={handleInputChange} />
      <button disabled={!state.isButtonEnabled}>제출</button>
      {state.isButtonEnabled && <ExpensiveComponent />}
    </div>
  );
};

const App = () => {
  return (
    <div>
      <h1>Form Example</h1>
      <Form />
    </div>
  );
};

스택 재조정자의 한계
스택 재조정자는 작업을 일시 중단하거나 연기하지 않고 동기적으로 순차적으로 변경 사항을 렌더링합니다. 이로 인해 다음과 같은 문제가 발생합니다:

  1. 렌더링 차단: 계산 비용이 비싼 컴포넌트가 렌더링을 완료할 때까지 메인 스레드가 차단되어, 사용자 입력이 눈에 띄게 버벅거리며 화면에 표시됩니다.
  2. 우선순위 부재: 모든 업데이트가 동일한 우선순위로 처리되므로, 중요한 사용자 상호작용(입력, 클릭 등)이 덜 중요한 시각적 업데이트에 의해 지연될 수 있습니다.
  3. 중단 불가능성: 일단 시작된 렌더링 프로세스는 완료될 때까지 중단할 수 없습니다. 즉, 더 높은 우선순위의 업데이트가 발생해도 진행 중인 렌더링을 중단하고 새로운 작업을 처리할 수 없습니다.

위 예제에서 사용자가 입력 필드에 타이핑할 때마다 ExpensiveComponent가 다시 렌더링되어 UI가 버벅거리고 사용자 경험이 크게 저하됩니다. 이상적으로는 사용자 입력에 즉시 반응하고, 덜 중요한 ExpensiveComponent의 렌더링은 나중에 처리하는 것이 좋을 것입니다. 이러한 문제를 해결하기 위해 React 팀은 작업을 작은 단위로 분할하고 우선순위를 부여할 수 있는 새로운 재조정 알고리즘이 필요하다고 판단했습니다. 그 결과 탄생한 것이 '파이버(Fiber)' 아키텍처입니다.

파이버 재조정자​

파이버 재조정자에는 조정자를 위한 작업 단위를 나타내는 파이버라는 데이터 구조가 사용됩니다. 파이버는 리액트 엘리먼트에서 생성되는데, 핵심적인 차이점은 파이버라는 상태를 저장하고 수명이 긴 반면 리액트는 엘리먼트는 임시적이고 상태가 업가 없다는 것입니다.

데이터 구조로서의 파이버​

리액트의 파이버 데이터 구조는 파이버 재조정자의 핵심 요소입니다. 파이버 재조정자는 업데이트의 우선순위를 정하고 이에 따라 동시 실행을 가능케 해서 리액트 애플리케이션의 성능과 응답성을 향샹시킵니다.

본질적으로 파이버 데이터 구조는 리액트 어플리케이션에서 컴포넌트 인스턴스와 그 상태를 표현합니다. 앞서 설명한 것처럼 파이버 데이터 구조는 변경 가능한 인스턴스로 설계되었으며 조정 과정에서 필요에 따라 업그레이드되고 배치됩니다. 파이버 노드의 각 인스턴스에는 해당 컴포넌트에 대한 정보, 다시 말해 프롭, 상태, 하위 컴포넌트 등이 포함됩니다. 뿐만 아니라 컴포넌트 트리 에서의 위치 정보, 파이버 재조정자가 업데이트 우선순위를 정하고 실행하는 데 사용하는 메타데이터도 들어 있습니다. 간단한 파이버 노드의 예시입니다.

{
  tag: 3, 
  type:  App,
  key: null, 
  ref: null, 
  props: {
    nane: "Tejas",
    age : 30
  },
  stateNode: AppInstance,
  return: FiberParent, 
  child: FiberChild,
  sibling: FiberSibling,
  index: 0
}

예시의 파이버 노드는 ClassComponent의 인스턴스를 나타냅니다. 각 필드는 다음과 같은 정보를 담고 있습니다

tag
예시에서는 3으로 설정되어 있는데 리액트는 이를 클래스 컴포넌트로 식별합니다. 각 컴포넌트 유형에는 고유한 숫자 ID가 파이버로 설정됩니다.

type
App은 이 파이버가 나타내는 함수 또는 클래스 컴포넌트를 나타냅니다.

props {name: "Tejas", age: 30}은 이 파이버에 전달된 프롭을 나타냅니다. 이 예시에서는 name과 age라는 두 개의 프롭이 있습니다.

stateNode
이 파이버가 나타내는 App 컴포넌트의 인스턴스입니다. 컴포넌트 트리에서의 위치는 return, child, sibling, index로 표현되는데 각각 부모, 자녀, 형제, 파이버의 색인을 의미하며, 파이버 재조정자는 이를 사용해 트리를 순회합니다.

파이버 재조정에는 현재 파이버 트리와 다음 파이버 트리를 비교해 어느 노드를 업데이트, 추가, 제거할지 파악하는 작업이 포함됩니다.

조정과정 중에 파이버 재조정자는 가상 DOM의 각 리액트 엘리먼트에 대해 파이버 노드를 생성합니다. 이 작업은 createFiberFromElement라는 함수에 의해 수행됩니다. 이 함수에서 말하는 '타입과 프롭'은 다른 말로 리액트 엘리먼트라고도 부를 수 있습니다. 리액트 엘리먼트는 타입과 프롭으로 이루어집니다.

{
  type: 'div',
  props: {
    className: "container"
  }
}

React에서 createFiberFromTypeAndProps 함수는 React 엘리먼트로부터 Fiber 노드를 생성하는 함수입니다. 이렇게 생성된 Fiber 노드는 이후의 재조정 과정에서 작업 루프(job loop) 를 통해 사용자 인터페이스(UI)를 업데이트하는 데 사용됩니다.

작업 루프란?

작업 루프는 루트 Fiber 노드에서 시작하여 전체 컴포넌트 트리를 순회합니다. 이 과정은 두 단계로 나뉩니다:

  1. 위에서 아래로: beginWork 단계
  • 각 컴포넌트를 평가하여 업데이트가 필요한지 확인합니다.
  • 필요한 경우 해당 Fiber 노드를 "더티(dirty)" 상태로 표시합니다.
  • "업데이트 대상"으로 플래그를 거는 작업입니다.
  1. 아래에서 위로: completeWork 단계
  • 다시 루트 방향으로 올라가며, 실제 DOM에 반영될 새로운 트리를 메모리상에서 생성합니다.
  • 이 DOM 트리는 브라우저와 분리된 상태(off-screen)이며, 사용자에게는 아직 보이지 않습니다.

이렇게 구성된 새 DOM 트리는 브라우저 화면에 즉시 나타나는 것이 아닙니다. 메모리 상에서만 존재하는 "오프스크린(off-screen)" 상태이기 때문에, 언제든지 중단하거나 폐기할 수 있습니다. 이 개념은 게임 개발에서 자주 사용하는 더블 버퍼링(double buffering) 기법에서 영감을 받았습니다.
→ 즉, 새로운 화면을 미리 준비해두고, 준비가 완료되었을 때 한 번에 현재 화면으로 전환하는 방식입니다.

더블 버퍼링​

더블 버퍼링은 컴퓨터 그래픽 및 비디오 처리에서 **깜빡임(flickering)**을 줄이고 체감 성능을 향상시키는 기술입니다. 이 기술은 두 개의 버퍼를 사용하여 이미지나 프레임을 번갈아 처리함으로써 부드러운 시각적 출력을 제공합니다.

  1. 버퍼 A에 초기 이미지나 프레임을 렌더링합니다.
  2. 버퍼 A가 화면에 표시되는 동안, 버퍼 B는 새 데이터나 이미지로 업데이트됩니다.
  3. 버퍼 B가 준비되면, 버퍼 A와 B가 전환되어 B의 내용이 화면에 표시됩니다.
  4. 이후 A와 B가 일정한 간격으로 교체되며, 렌더링이 계속 진행됩니다.
이렇게 하면 중간 상태가 보이지 않으므로 깜빡임이나 시각적 잔상 없이 부드러운 화면을 제공할 수 있습니다.

React의 파이버(Fiber) 재조정(reconciliation) 과정은 더블 버퍼링과 개념적으로 유사합니다.

  1. 업데이트가 발생하면, 현재의 파이버 트리를 복제한 '작업용(workInProgress)' 트리가 생성됩니다.
  2. 이 작업용 트리는 새로운 UI 상태를 반영하도록 업데이트되며, 이를 **렌더링 단계(render phase)**라고 부릅니다.
  3. 작업이 완료되면, 작업용 트리와 현재 트리를 교체하여 UI를 갱신합니다. 이 과정을 커밋(commit) 단계라고 합니다.
파이버 재조정의 장점
  1. 시각적 일관성: 중간 DOM 업데이트를 피하여 깜빡임 및 불완전한 렌더링을 방지합니다.
  2. 업데이트 우선순위 조정: UI의 새 상태를 계산 중에도 더 중요한 업데이트가 발생하면 작업을 중단하고 우선 수행할 수 있습니다.
  3. 작업 중단 및 재개: 사용자에게 영향을 주지 않고, 렌더링을 일시 중지 및 재시작할 수 있습니다.
  4. 점진적 렌더링: 큰 작업을 작은 단위로 분할하여, 메인 스레드의 차단을 최소화할 수 있습니다.

React는 JSX 엘리먼트 트리로부터 두 개의 파이버 트리를 생성합니다

  • current 트리: 현재 화면에 표시되고 있는 상태
  • workInProgress 트리: 다음 화면 상태를 반영하기 위해 백그라운드에서 준비 중인 트리

이 두 트리 사이의 교체 작업이 더블 버퍼링에서의 버퍼 전환과 유사합니다.

파이버 재조정​

파이버 재조정은 크게 렌더링 단계와 커밋 단계로 나뉩니다.

렌더링 단계
렌더링 단계는 현재 트리에서 상태 변경 이벤트가 발생하면 시작합니다. 리액트는 각 파이버 노드를 순회하면서 업데이트가 필요한지 확인하고, 필요한 경우 작업용 트리에 해당 변경사항을 적용합니다. 이 과정은 리액트 내부적으로 beginWork와 completeWork 함수를 통해 이루어집니다.

beginWork(작업 시작)
beginWork 함수는 작업용 트리의 파이버 노드를 검사하고 업데이트가 필요한지 결정합니다. 필요한 경우 해당 노드에 업데이트 플래그를 설정하고, 자식 노드들로 재귀적으로 이동하며 동일한 과정을 반복합니다.

function beginWork(
  current: Fiber | null, 
  workInProgress: Fiber,
  renderLanes: Lanes, 
): Fiber | null;

함수 매개변수의 의미

  • current: 현재 트리에서 업데이트 중인 작업용 노드에 대응하는 파이버 노드 참조입니다. 이전 상태와 새 상태 간의 차이점을 파악하는 데 사용됩니다.
  • workInProgress: 작업용 트리에서 현재 처리 중인 파이버 노드입니다. 이 노드가 업데이트되어 반환됩니다.
  • renderLanes: 업데이트의 우선순위를 나타내는 비트마스크입니다. React 17부터 도입된 개념으로, 이전의 renderExpirationTime을 대체합니다.
renderLanes와 우선순위

renderLanes는 여러 업데이트를 우선순위에 따라 분류하는 방식입니다. 리액트에서 컴포넌트 변경이 발생하면 해당 변경의 중요도와 긴급성에 따라 다른 '레인'이 할당됩니다

  • 사용자 입력과 같은 상호작용은 높은 우선순위(높은 레인)
  • 데이터 로딩과 같은 백그라운드 작업은 낮은 우선순위(낮은 레인) 이 메커니즘을 통해 리액트는 중요한 업데이트를 먼저 처리하고 덜 중요한 작업은 나중에 처리할 수 있습니다.

completeWork(작업 완료)
트리의 말단 노드에 도달하면 completeWork 함수가 호출됩니다. 이 함수는 변경된 노드에 대해 실제 DOM 조작을 준비합니다.

function completeWork() {
  current: Fiber | null,
  workInProgress: Fiber,
  renderLanes: Lanes,
}: Fiber | null;

completeWork는 다음과 같은 작업을 수행합니다

  1. 호스트 컴포넌트(DOM 요소)의 경우 필요한 DOM 노드를 생성하거나 업데이트합니다.
  2. 속성 변경, 스타일 업데이트, 이벤트 리스너 설정 등을 준비합니다.
  3. 이 모든 변경사항을 아직 실제 DOM에 적용하지는 않고, 커밋 단계를 위해 준비만 합니다.

렌더링 단계에서는 실제 DOM 변경이 일어나지 않으며, 모든 작업은 메모리상에서만 이루어집니다. 이는 작업을 중단하거나 폐기할 수 있게 해줍니다.

커밋 단계
렌더링 단계가 완료되면 커밋 단계가 시작됩니다. 이 단계에서는 렌더링 단계에서 계산된 변경사항을 실제 DOM에 적용합니다. 커밋 단계는 중단할 수 없으며 동기적으로 실행됩니다. 커밋 단계는 크게 세 부분으로 나뉩니다

변형 전 단계
DOM 변경 전에 필요한 준비 작업을 수행합니다. 스냅샷을 생성하거나 getSnapshotBeforeUpdate 생명주기 메서드를 호출합니다.

변형 단계
실제 DOM 변경이 이루어지는 단계입니다. commitMutationEffects 함수가 호출되어 다음 작업을 수행합니다

  • DOM 노드 추가, 업데이트, 삭제
  • 텍스트 콘텐츠 변경
  • 속성 및 스타일 업데이트
  • 이벤트 리스너 설정/제거

이 단계에서는 useLayoutEffect 훅의 정리(cleanup) 함수도 실행됩니다.

레이아웃 단계
DOM 변경이 완료된 후 실행되는 단계입니다. commitLayoutEffects 함수가 호출되어 다음 작업을 수행합니다:

  • componentDidMount와 componentDidUpdate 생명주기 메서드 호출
  • useLayoutEffect 훅의 효과 함수 실행
  • 레이아웃 관련 계산 수행

레이아웃 단계가 완료되면 작업용 트리가 현재 트리가 되고, 이전 현재 트리는 재사용을 위해 보관됩니다.

효과
리액트 재조정 과정의 커밋 단계에서는 여러 부작용이 특정 순서로 실행되며, 그 순서는 효과 종류에 따라 달라질 수 있습니다. 커밋 단계에서 발생하는 효과는 다음과 같습니다.

  1. 배치 효과 (Placement Effects)
  • 새로운 DOM 노드를 화면에 추가하는 작업을 수행합니다
  • 컴포넌트가 처음 마운트될 때 DOM에 요소를 삽입합니다
  • Placement 플래그가 있는 Fiber 노드에 대해 실행됩니다
  1. 업데이트 효과 (Update Effects)
  • 이미 존재하는 DOM 노드의 속성, 스타일, 이벤트 핸들러 등을 업데이트합니다
  • Update 플래그가 있는 Fiber 노드에 대해 실행됩니다
  • 컴포넌트의 props나 state가 변경되었을 때 발생합니다
  1. 삭제 효과 (Deletion Effects)
  • DOM에서 노드를 제거하는 작업을 수행합니다
  • 컴포넌트가 언마운트될 때 발생합니다
  • 관련된 이벤트 리스너와 참조를 정리합니다
  • Deletion 플래그가 있는 Fiber 노드에 대해 실행됩니다
  1. 레이아웃 효과 (Layout Effects)
  • DOM 변경 직후, 브라우저가 화면을 다시 그리기 전에 실행됩니다
  • useLayoutEffect 훅으로 정의됩니다
  • DOM 측정이 필요하거나 화면에 시각적인 변화를 즉시 적용해야 할 때 사용합니다
  • 동기적으로 실행되므로 렌더링을 차단할 수 있습니다
  1. 패시브 효과 (Passive Effects)
  • 브라우저의 페인팅(화면 그리기) 후에 비동기적으로 실행됩니다
  • useEffect 훅으로 정의됩니다
  • 데이터 페칭, 이벤트 구독, 타이머 설정 등과 같은 비동기 작업에 적합합니다
  • 화면 렌더링을 차단하지 않아 성능에 더 유리합니다
  • 정리(cleanup) 함수를 반환하여 컴포넌트 언마운트 시 리소스를 해제할 수 있습니다
  • 의존성 배열에 따라 선택적으로 실행됩니다
